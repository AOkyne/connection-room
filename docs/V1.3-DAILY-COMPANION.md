# v1.3 Daily Companion System

## Overview

The Connection Room v1.3 introduces a "Daily Companion" experience that transforms the dashboard from a community bulletin board into a personalized daily ritual. Members arrive to find their name, today's theme, a reflection prompt, embodiment practice, body check-in, conversation invitation, a featured quote, and Trevor's weekly note.

## Core Concept

Instead of "What content is available?" the dashboard asks: "What would help this member feel a little more connected today?"

The experience is:
- **Gentle**: No pressure, no gamification, no streaks
- **Grounded**: Embodiment, sensation, presence-based practices
- **Private**: Reflections are saved privately by default
- **Rotational**: Content cycles every 120 days, with weekly notes cycling every 16 weeks

## Architecture

### Database Schema

#### `daily_companion_content`
Stores all daily content that rotates daily.

```sql
CREATE TABLE daily_companion_content (
  id TEXT PRIMARY KEY,
  content_type TEXT, -- 'theme', 'reflection', 'practice', 'checkin', 'invitation', 'quote'
  title TEXT,
  body TEXT,
  category TEXT,
  intended_date DATE, -- optional override
  rotation_index INTEGER, -- 0-119 (maps to day of cycle)
  active BOOLEAN,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

**Content Types:**
- `theme`: Daily theme/intention (120 items)
- `reflection`: Daily reflection prompt (120 items)
- `practice`: Embodiment practice (120 items)
- `checkin`: Body check-in prompt (120 items)
- `invitation`: Conversation invitation (120 items)
- `quote`: Daily quote (120 items)

#### `weekly_notes`
Stores weekly letters from Trevor, one per week.

```sql
CREATE TABLE weekly_notes (
  id TEXT PRIMARY KEY,
  week_number INTEGER, -- 1-16
  title TEXT,
  body TEXT, -- ~150-200 words
  related_prompt_id TEXT, -- optional reference to daily content
  related_space_id TEXT, -- optional space to explore
  intended_date DATE, -- optional override
  rotation_index INTEGER, -- 0-15 (maps to week of cycle)
  active BOOLEAN,
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

#### `user_reflections`
Stores private user reflections.

```sql
CREATE TABLE user_reflections (
  id TEXT PRIMARY KEY,
  user_id UUID,
  content_id TEXT, -- reference to daily_companion_content
  prompt_text TEXT, -- snapshot of prompt at time of reflection
  response TEXT,
  privacy_setting TEXT, -- 'private', 'shared' (future)
  created_at TIMESTAMP,
  updated_at TIMESTAMP
);
```

RLS Policies:
- Users can only view/edit/delete their own reflections
- Daily companion content is read-only to all authenticated users
- Reflections are private by default

#### Key Functions

**`getTodaysDailyContent()`**
Returns today's daily content (theme, reflection, practice, check-in, invitation, quote).

```typescript
const daily = await getTodaysDailyContent();
// Returns: {
//   theme: DailyContent,
//   reflection: DailyContent,
//   practice: DailyContent,
//   checkin: DailyContent,
//   invitation: DailyContent,
//   quote: DailyContent
// }
```

Uses `getDaysSinceLaunch() % 120` to determine which content to show.

**`getTrevorWeeklyNote()`**
Returns this week's Trevor note.

```typescript
const note = await getTrevorWeeklyNote();
// Returns: WeeklyNote | null
```

Uses `getWeekSinceLaunch() % 16` to determine which note to show.

**`saveUserReflection(userId, contentId, promptText, response)`**
Saves a user's reflection (private by default).

```typescript
await saveUserReflection(userId, contentId, prompt, response);
```

**`getUserReflections(userId, limit)`**
Fetches user's past reflections (most recent first).

```typescript
const reflections = await getUserReflections(userId, 5);
```

**`updateUserReflection(reflectionId, response)`**
Updates an existing reflection.

```typescript
await updateUserReflection(reflectionId, newResponse);
```

**`deleteUserReflection(reflectionId)`**
Deletes a reflection.

```typescript
await deleteUserReflection(reflectionId);
```

**`seedDailyCompanionContent()`**
Seeds all daily content and weekly notes to database. Run once during initial setup.

```typescript
await seedDailyCompanionContent();
```

## How Daily Content Rotates

### Algorithm

Content rotates based on **days since app launch** (January 1, 2024):

```typescript
const launchDate = new Date("2024-01-01");
const today = new Date();
const daysSinceLaunch = Math.floor((today - launchDate) / (1000 * 60 * 60 * 24));
const todayContentIndex = daysSinceLaunch % 120;
```

This means:
- Day 0-119: First 120 items
- Day 120-239: Second cycle (same items, different 120 days)
- Day 240+: Third cycle, etc.

Members always see the same content on the same date. New members arriving in week 50 see the same theme/reflection as members who were there on day 50.

### Timezone Handling

All content is determined by UTC date. Admins can override with `intended_date` field if needed.

### Weekly Rotation

Weekly notes cycle every 16 weeks:

```typescript
const weeksSinceLaunch = Math.floor(daysSinceLaunch / 7);
const weekContentIndex = weeksSinceLaunch % 16;
```

Week 16 is a "coming home" final note that loops back to week 0.

## Demo Mode

When `supabase` is not available or in development, the system uses:

1. **Seed data arrays**: All 120 themes, reflections, etc. are available locally
2. **localStorage**: Reflections are saved to `localStorage["demo-reflections"]`
3. **No database queries**: All functions return objects constructed from seed arrays

This allows the daily companion to work completely offline in demo mode.

## How to Extend

### Adding More Content Cycles

Currently: 120 days × 4 months = 1-year cycle

To extend (e.g., 240-day cycle):
1. Add 120 more items to each array in `lib/seed/daily-companion-content.ts`
2. Update arrays: `dailyThemes`, `reflectionPrompts`, `embodimentPractices`, etc.
3. No code changes needed; rotation algorithm scales automatically

### Editing Content

To edit existing content:

1. **Seed data**: Edit the arrays in `lib/seed/daily-companion-content.ts`
2. **Database**: Create a migration to update `daily_companion_content` table
3. **Or use admin interface** (future feature)

import { initializeDailyCompanion } from "@/lib/seed/init-daily-companion";

useEffect(() => {

```bash

```

Create a migration file to apply `lib/supabase/schema-v1.3.sql`:

```bash
# In Supabase console, run schema-v1.3.sql
# Or via CLI:
supabase migration new add_daily_companion

```

## Performance Considerations

### Query Optimization

- All queries use `rotation_index` and `active` columns with indexes
- `getTodaysDailyContent()` runs 6 queries (one per content type) but results are cached in memory
- No N+1 queries; all data loaded upfront

### Caching

Consider caching daily content for 24 hours:

```typescript
const CACHE_KEY = `daily-content-${getTodaysDailyContent().rotation_index}`;
const cached = localStorage.getItem(CACHE_KEY);
```

### Component Optimization

All daily companion components use:
- `memo` for preventing re-renders
- Lazy loading where applicable
- Progressive enhancement (works without JavaScript)

## Privacy & Security

### RLS Policies

- **Daily content**: Public read to all authenticated users
- **User reflections**: Private to user (user can only access own reflections)
- **Weekly notes**: Public read to all authenticated users

No unauthenticated access. All data is behind auth wall.

### Data Handling

- Reflections never auto-shared
- Privacy setting defaults to `'private'`
- No analytics on reflection content
- Users can delete reflections at any time
